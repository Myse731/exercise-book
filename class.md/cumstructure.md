# 데이터 표현의 기초

## 컴퓨터는 왜 0과 1만 쓸까?

#### 컴퓨터는 전기로 작동하는 기계로 복잡한 10진수를 사용하는 것보다 전압이 높음과 낮음 두 상태만 구분하는 것이 훨씬 정확하고 빠르다

## 데이터 단위

## 비트

- 0 과 1을 표현하는 가장 작은 단위
- n비트로 표현할 수 있는 정보의 가짓수는 2^n
- 1비트가 추가될수록 표현 가능한 정보는 2배씩 늘어난다

## 단위

```python
1. Bit(비트)

* 컴퓨터가 처리하는 가장 작은 단위
* 0 또는 1 두가지 상태만 가질수 있다

2. Byte(바이트)

* 비트 8개를 하나로 묶은 단위
* 2^8 = 256가지의 정보를 표현할 수 있어, 영문자 한 글자를 저장할수있는 기본 단위가 됩니다.
```

```python
컴퓨터는 10진수 1000 가장 가까운 2진수인 2^10(=1,024)을 기준으로 단위를 올린다

1 Byte
| x 1,000
KB(KiloByte) : 2^10 Byte
| x 1,000
MB(MegaByte) : 2^10 KB
| x 1,000
GB(GigaByte) : 2^10 MB
| x 1,000
TB(TeraByte) : 2^10 GB

---- 1000 이랑 1024랑 같다고 보는게 괜찮을까?----

용량이 줄어든 것은 아니고, 숫자를 세는 기준(단위)이 달라서 생기는 현상
최신 표준은 혼란을 막기 위해 1,024 단위를 KiB, MiB, GiB(키비, 메비, 기비)라고 부르기로 함.

단, 정보처리기능사 같은 기초 자격증 시험에는 여전히 1,024B = 1KB로 봄.
```

## 16진수

- 한 자리에 0~15까지의 숫자로 수를 표현
- 2진수는 너무 길고 10진수는 컴퓨터와 친하지 않기 때문에 개발자는 16진수를 활용
- 활용 예시 : 웹 디자인, 메모리 주소, 네트워크, 임베디드 레지스터 제어
- 인간의 인지 오류 감소, 메모리 구조의 직관적 파악, 하드웨어 상태 시각화, 문서화 및 통신 효율성

### 사용이유 : 데이터 가독성과 압축률

- 32비트 데이터를 표현할 때, 2진수는 32개 숫자가 필요하지만 16진수는 단 8개면 충분하다.
4:1의 압축비가 엔지니어들이 16진수를 사용하는 실질적인 이유이다.

### 2진수 → 16진수

```python
1. 4자리 나누기

1101 | 0110 #뒤에서부터 끊어줌

2. 10진수 계산

1101 = 13

0110 = 6

3. 16진수 매핑
13 -> D
6 -> 6
'D6'
```

### 16진수 → 2진수

## 이진수의 덧셈과 뺄셈

- 덧셈
    
    ```css
    1001 + 1101
    
    = 10111
    
    1100 + 10110
    
    = 100010
    ```
    
- 뺄셈
    
    ```css
    기본 규칙
    덧셈기와 뺄셈기를 모두 만들면 컴퓨터 칩이 크고 복잡해짐. 보수를 이용하면
    기존의 덧셈기만으로 뺄셈을 완벽하게 수행할 수 있어 효율적
    '부족한 수를 채워주는 수'라는 뜻
    ex) 3에 대한 10의 보수 : 7, 4에 대한 10의 보수 : 6
    수학적으로 뺄셈을 덧셈으로 바꾸는 도구이다. ex) 15 - 3
    1. 10에 대한 3의 보수 : 7
    2. 15+7 = 22
    3. 올림수 10을 빼고 남은수 : 12
    
    2진수에서의 보수
    • 1의 보수 : 모든 비트를 반대로 뒤집은 수
     (1010 -> 0101) -> "0과 1을 서로 보충해주는 관계)
     
    2의 보수 : 1의 보수에 1을 더한 수 -> "컴퓨터가 사용하는 진짜 음수"
    ```
    

## 부호의 표현과 데이터 한계

- 4비트 부호 범위 : -8 ~ 7
- 5비트 부호 범위 : -16 ~ 15
- n비트 부호 범위 : -2^n ~ 2n^n-1 -1

```css
만약 최대 7(0111)인 값에 1(0001)을 더하면 어떻게 될까?

= 1000(8)

양수 + 양수 = 음수

양수 + 양수를 했는데 음수가 되거나 음수 + 음수를 했는데 양수가 나오는 경우
즉, 계산 겨로가가 부호비트 칸을 침범하는 현상을 오버(언더 플로우)라고 한다.
```

# 레지스터

- PC(Program Counter, 프로그램 카운터)
    - 역할 : CPU가 다음에 실행할 명령어의 메모리 주소를 가리키는 이정표 역할을 합니다.
    - 특징 : 명령어가 하나씩 실행할 때마다 값이 자동으로 증가하여 다음 단계로 안내합니다. 만약 ‘점프’나 ‘루프’ 같은 명령어가 실행되면 해당 주소로 값이 직접 바뀝니다.
- AR(Address Resigter, 주소 레지스터)
    - 역할 : CPU가 데이터를 읽거나 쓰기 위해 접근하려는 메모리의 주소를 일시적으로 보관합니다.
    - 특징 : PC 로부터 주소를 전달받아 “메모리의 몇 번지 데이터를 가져올지”를 주소 버스에 실어 보내는 관문입니다.
- DR(Data Registar, 데이터 레지스터)
    - 역할 : 메모리와 CPU 사이에서 주고받는 실제 데이터를 임시 저장하는 레지스터이다.
    - 특징 : 메모리 참조 명령에서 주로 사용되며, ALU의 피연산자로도 사용된다.
- IR(Instruction Register, 명령어 레지스터)
    - 역할 : 메모리에서 가져온 현재 실행 중인 명령어 자체를 담고 있습니다.
    - 특징 : 제어장치는 이 IR에 담긴 비트를 해독하여 ALU에게 어떤 연산을 할지, 메모리에서 어떤 데이터를 더 가져올지 지시를 내립니다.
- AC(Accumulator, 누산기)
    - 역할 : 연산의 중간 결과나 최종 결과를 일시적으로 보관하는 저장소 입니다.
    - 특징 : 이름처럼 값을 ‘누적’하는 곳입니다. 예를 들어 A + B 를 할 때, A를 먼저 담아두고 B가 오면 ALU와 함께 더한 뒤 그 결과값을 다시 자신의 자리에 덮어씁니다.

# 명령어 실행 사이클과 주소 지정 방식

## 주소 모드

### 묵시적 주소 지정 방식

```css
명령어 안에 피연산자(operand)의 주소를 직접 쓰지 않아도,
CPU가 사용할 연산자의 위치를 이미 알고 있는 방식
CMA -> AC의 비트를 반전하라
```

### 즉시 주고 지정 방식

```css
피연산자가 명령어 자체 내에 포함 되어 있는 형태로 값 자체가 
그대로 데이터이다.
LOAD #5 <- 5를 보고 주소를 찾아가는게 아니라 
AC = 5이다.(#은 값이라는 의미이다)
```

### 직접 주소 지정 방식

!image.png

```css
개념 명령어의 주소부 A가 실제 데이터의 주소
표기 : LOAD A
EA(Effective Address = 실제로 접근할 메모리 주소)
EA = A
AR : AR <- EA
```

### 간접 주소 지정 방식

!image.png

```css
개념 : A가 가리키는 곳에 "진짜 주소"가 있음
** 명령어의 주소부만으로는 전체 메모리 공간을 직접 저장할 수 없을때
사용하며, 포인터 같은 간접 참조를 지원

표기 : LOAD @ A 또는 LOAD(A)

AR :
AR <- A
AR <- M[AR] (AR <- EA)
```

### 상대 주소 지정 방식

!image.png

```css
개념 : 현재 PC 기준으로 주소 계산
** 프로그램을 메모리 어디에 올려도 실행 가능하게 하려고 사용**

표기 : BEQ A
EA : EA = PC + A
AR : AR <- EA
```

### 베이스 레지스터 주소 지정 방식

!image.png

```css
개념 : 기준 주소(Base) + offset
** 프로그램이나 데이터 영역의 시작 주소를 넣어둔다.
주소부가 짧아도 넓은 메모리 공간이 접근 가능하다
 + 추가 메모리 접근 없음 **
 
표기 : LOAD A(RB)
EA : EA = RB + A
AR : AR <- EA
```

### 인덱스 주소 지정 방식

```css
개념 : Index + offser[배열 접근]

표기 : LOAD A(IX) 또는 A[i]

EA : EA = IX + A
AR : AR <- EA
활용 : 배열, 테이블 탐색
```
